## 2. Проверка

```bash
bash scripts/check.sh
```

Если менялся backend:

- убедиться, что SQLite backup будет создан перед деплоем;
- проверить, что новые env-переменные описаны в `.env.example`, но реальные значения не попали в Git.
- если добавлялись author-таблицы, проверить production SQLite после рестарта API.
- если менялась модерация авторов, проверить `recipe_publications`, `recipe_publication_versions`, `recipe_publication_events` и обратную совместимость старых публикаций.
- если менялись Telegram-уведомления авторов, проверить `.env.example`, webhook `@MBS_work_bot`, привязку автора и отсутствие реальных токенов в Git.
- после production-деплоя Telegram-уведомлений проверить, что `/api/telegram/join-mbs/webhook` установлен на `@MBS_work_bot`, `@Join_MBS_bot` не занят платформой и автор может подключить Telegram из кабинета.
- если менялся Mixology auto-author, проверить, что приватный whitelist не попал в Git и что доступ выдаётся только по `visited`.
- если менялся блок чемпионатов в профиле автора, проверить `author_championship_participations`, `GET /api/author/profile` и отсутствие приёма этих данных в `PUT /api/author/profile`.

Если менялась admin-панель:

- менять исходники `server/admin/src/*`;
- после этого обязательно собрать `server/admin/admin-panel.js`;
- деплоить именно bundle.
- проверить light/dark layout: фон занимает всю ширину Tilda, `#adm-panel` центрирован, нет белых полей по бокам в тёмной теме.

Если менялся frontend:

- проверить desktop/mobile навигацию;
- проверить доступы `drinks`, `finance`, `author`;
- проверить no-access экран и первую доступную вкладку.
- для author mode проверить, что обычный клиентский режим `Напитки` не потерял полный набор рецептов/поставщиков.
- если менялись `План продаж` или `Финмодель`, проверить на desktop и mobile: таблицу напитков, строку `ИТОГО`, блок `Дополнительные продажи`, горизонтальный скролл таблицы доп. продаж, KPI финмодели, P&L, What-if.
- если менялся блок постоянных расходов в финмодели, проверить: раскрытие категорий, добавление/редактирование расхода, переключение `Сумма ₽/мес` / `% от выручки`, подсказки summary-плашек `ручные` / `ФОТ` / `масштаб.`, переход из авто-ФОТ к калькулятору ФОТ.
- если менялась командная работа, проверить: создание личного workspace из старого state для платного пользователя, запрет guest создавать свои проекты, приглашение по email/link, регистрацию по invite-link, принятие приглашения вторым пользователем, запрет editor управлять командой, общий state проекта и записи в `Журнале действий` без шума autosave.
- если менялся `Журнал действий`, проверить: повторный вход не создаёт много одинаковых `Вход в проект`, фильтры `Все/Команда/Заведения/Бюджет/Финмодель/Рецепты/Поставщики/Экспорт/Безопасность` работают, в строке видна роль участника, важные/опасные события визуально отличаются, summary бюджета открытия содержит категорию/название/сумму, заблокированная критическая попытка guest/editor отображается как `Действие заблокировано`.
- если менялись точки восстановления workspace, проверить: owner видит `Восстановление`, может создать ручную точку, autosnapshot появляется после изменения state, restore возвращает проект к снимку и создаёт `before_restore`, editor/guest не может восстановить проект.
- если менялись structural/destructive-действия в workspace, проверить: owner может добавить/создать из шаблона/переименовать/удалить заведение; editor/guest не видит или получает блокировку на эти действия; owner может удалить рецепт/поставщика/сырьё/полуфабрикат; editor/guest не видит или получает блокировку на эти удаления; owner видит верхний `Сброс` и `Очистить всё` в бюджете, editor/guest не видит или получает блокировку; прямой `PUT /api/state` от editor/guest с появившимися/исчезнувшими `locIndex`/`locations`, изменённым metadata `locIndex`, исчезнувшими `customDrinks`/`customMats`/`semiItems`/`suppliers`/`supplierBook`, очищенными важными списками или массовым reset-sensitive diff возвращает `403` и пишет `state_update_blocked`; обычное редактирование содержимого существующей точки у editor/guest продолжает работать.
- если менялись финансовые расчёты, проверить, что `Выручка`, `Прибыль`, `FC%`, `Средний чек`, `ТБУ`, `Окупаемость` и `Запас прочности` используют общий план продаж: напитки + дополнительные позиции.
- если менялись отчёты, проверить `PDF — полный отчёт`, `Excel (xlsx)` и `Скачать CSV`; PDF не должен содержать блок `Рейтинг напитков`, Excel может сохранять рейтинг на dashboard.

## 3. Production smoke

```bash
npm run smoke:api
```

Для теста выдачи доступа автора и Битрикс-синхронизации:

```bash
npm run smoke:api:apply
```

Smoke проверяет:

- `https://barista-school.online/api/health`;
- admin login;
- `/api/auth/me` возвращает `access`;
- `/api/admin/users` возвращает access-флаги;
- тестовый пользователь найден по телефону/email;
- `author_profiles` существует;
- `bitrix_contact_id` совпадает с ожидаемым, если включён `expect_bitrix_synced`;
- public API не отдаёт приватные поля.

## 3.1. Production network smoke

После frontend/backend/nginx-деплоя проверить:

```bash
curl -Iv --http2 https://barista-school.online/
curl -fsS --http2 https://barista-school.online/api/health
```

Ожидаемо:

- `ALPN: server accepted h2`;
- главная отвечает `HTTP/2 200`;
- `/api/health` отвечает `{"ok":true,"version":"1.0.0"}`.

Если сайт в РФ открывается только через VPN:

- сначала проверить HTTP/2 на nginx, а не frontend-кэш;
- проверить, что в `/etc/nginx/sites-enabled/` нет backup-файлов с server block;
- не хранить backup внутри `sites-enabled/*`, потому что nginx включает все файлы wildcard-ом;
- сверить `nginx -t` без warning `conflicting server name`;
- для соседних Beget/VPS-хостов проверить тот же симптом: `curl -Iv --http2 https://host/` не должен оставаться на `HTTP/1.1`.

## 5. Публичные данные

Перед публикацией витрины проверить, что public API не отдаёт:

- телефон;
- email;
- Telegram;
- yClients ID;
- `author_user_id`, себестоимость, review-комментарии и полный `recipe` snapshot;
- внутренние ID, которые не нужны покупателю.

Для поставщиков:

- anonymous `GET /api/suppliers` должен требовать авторизацию;
- авторизованный клиент может получать публичные телефоны поставщиков;
- `GET /api/admin/suppliers` остаётся только для admin.

Последняя проверка 22 июня 2026 после security-deploy: anonymous `GET /api/suppliers` вернул `403 Not authenticated`, Роман подтвердил, что авторизованный клиент видит телефоны поставщиков.

Для простой public-витрины проверить:

- `/recipes` показывает список опубликованных рецептов;
- `/recipes/{slug}` показывает страницу рецепта без полной технологии приготовления;
- `POST /api/public/author-recipes/{recipe_id}/order` создаёт заявку на один рецепт.

## 5.1. Безопасность публичных endpoint'ов

- проверить, что `/api/telegram/setup` недоступен публично; использовать только `/api/admin/telegram/setup` с admin JWT;

- проверить, что `/api/proxy-meta` отклоняет `localhost`, private IP, `.local`, URL с логином/паролем и нестандартные порты;
